Docker is an open platform for developing, shipping, and running applications. Docker enables you to separate your applications from your infrastructure so you can deliver software quickly.

### **Container**
Docker provides the ability to package and run an application in a loosely isolated environment called a container

The isolation and security lets you run many containers simultaneously on a given host. Containers are lightweight and contain everything needed to run the application, so you don't need to rely on what's installed on the host.

### Why are Containers Lightweight?
**They don't carry their own Operating System.**

**Virtual Machines (The "Heavy" Way):** Imagine you want to run 3 apps using VMs. You need 3 separate Virtual Machines. Each VM has to load a full Operating System (Guest OS), which needs its own CPU, RAM, and storage just to exist.

**Containers (The "Light" Way):** Containers sit on top of your existing OS (Linux/Windows). They share the **Host OS Kernel** (the core brain of the computer). They only package the application and the tiny bits of code needed to run it.


## Why containers are called **OS-based VMs**

- They provide **process isolation**
- They give the illusion of a separate OS
- Each container has:
    - Its own filesystem
    - Its own network namespace
    - Its own process tree
    - Its own user IDs

But under the hood:

- Same kernel
- Same OS
- Same hardware

## Dockers characterstics -
#### **1. Portability ("Write Once, Run Anywhere")**

- **The Concept:** This is Docker's biggest selling point. It bundles your code (Nestibly), runtime (Node.js), libraries, and settings into a single package (Image).
    
- **Why it matters:** It solves the "It works on my machine" problem. If it runs on your laptop, it guarantees it will run on an AWS EC2 instance, a colleague's computer, or a production server exactly the same way.
    

#### **2. Isolation**

- **The Concept:** Every container has its own resources that are invisible to other containers.
    
- **Why it matters:** You can run a Node.js v14 app and a Node.js v18 app on the same server at the same time without them conflicting or crashing each other. They don't know the other exists.
    

#### **3. Lightweight & Fast**

- **The Concept:** As we discussed, they share the host OS kernel and don't need a full Guest OS.
    
- **Why it matters:** You can start a Docker container in milliseconds (ms), whereas a Virtual Machine takes minutes to boot up. This makes them perfect for scaling up quickly when traffic spikes.
    

> [!NOTE]
> #### **4. Immutability (Unchangeable)**
> 
> - **The Concept:** Docker images are "Read-Only." Once you build an image, it cannot be changed. If you want to update your code, you create a _new_ image.
>     
> - **Why it matters:** This creates consistency. You never have to worry that someone manually logged into the server and changed a file that broke the app. The image is the single source of truth.

    

#### **5. Version Control (For Infrastructure)**

- **The Concept:** Just like you use Git for your code, Docker allows you to version your environments.
    
- **Why it matters:** You can track changes to your `Dockerfile`. If version 2.0 of your app breaks, you can instantly rollback to the Docker Image for version 1.9.
    

---

### **3. Advanced Characteristic: The "Layered File System"**

_This is a "bonus point" answer._

Docker uses a **Union File System**. This means an image is built in layers (like a cake).

- **Layer 1:** The Base OS (Alpine Linux).
    
- **Layer 2:** Node.js installation.
    
- **Layer 3:** Your `package.json`.
    
- **Layer 4:** Your App Code.
    

**Benefit:** If you change your code (Layer 4), Docker only rebuilds Layer 4. It reuses Layers 1-3 from the cache. This makes building and sharing images incredibly fast.
